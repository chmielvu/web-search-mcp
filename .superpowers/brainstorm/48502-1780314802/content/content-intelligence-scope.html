Content Intelligence Pipeline: Scope Choice

Current evidence: the repo already has explicit fetch/search contracts, ContentArtifact.quality_score, classify_markdown gates, LanceDB semantic cache with hybrid/RRF, and weighted RRF merge for web results.

Validated Current Surface

What should be the first design center of gravity?

A

Fetch-First Enrichment

Make get_content the primary intelligence entrypoint: quality, entities, schema extraction, and optional KB save happen only when explicitly requested per fetch.

B

KB-First Memory

Design persistent saved-page storage and retrieval first, then attach quality/entities/extraction as metadata that makes KB search better.

C

Search-Fusion First

Start from web_search + search_kb fusion, using quality-weighted RRF to blend fresh web hits and saved KB hits in one result flow.
