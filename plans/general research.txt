2. Shortlist recommendations
Best default for agentic workflows: Tavily

Tavily is still the most commonly praised “agent-ready” option: search, extract, map, crawl, and research are exposed as API products, and the pricing is easy to prototype with: 1,000 free credits/month, basic search at 1 credit, advanced search at 2 credits, and pay-as-you-go at $0.008/credit.

Community praise is mainly about clean JSON, lower integration effort, and LLM-oriented extracted content, not necessarily absolute search quality. In benchmarks, Tavily was fast enough for interactive agents, but it ranked below Brave/Firecrawl/Exa/Parallel in one 2026 benchmark.

Use Tavily when: you want a working agent quickly, especially with LangChain/CrewAI/LlamaIndex-style stacks.

Avoid relying on Tavily alone when: you need reproducible SERP parity with Google, exact SEO-style results, or very high-volume low-cost search.

Best raw web-search API: Brave Search API

Brave is strong if you want a real search index, low latency, and control over downstream processing. Brave’s Search plan is $5/1,000 requests, includes $5 free credits/month, supports 50 QPS, and returns URLs, text, news, images, plus LLM-optimized context; its Answers plan is $4/1,000 requests + $5/M input/output tokens and supports citations/streaming/OpenAI SDK compatibility.

AIMultiple’s May 2026 benchmark ranked Brave highest by “Agent Score,” with 669 ms average latency, while noting that Brave, Firecrawl, Exa, and Parallel Pro were statistically close in the top tier.

Use Brave when: you want fast discovery, an independent index, and control over extraction/reranking.

Pitfall: Brave is more “raw search” than “agent context package.” You will often need Firecrawl, Jina, or your own fetch/extract pipeline.

Best semantic/code/docs search: Exa

Exa is strongest when the query is conceptual rather than literal. Its pricing page positions Search at $7/1,000 requests, Contents at $1/1,000 pages per content type, Deep Search at $12–15/1,000 requests, and Answer at $5/1,000 requests; it also offers 1,000 free requests/month.

Its practical advantage is not “Google but cheaper”; it is semantic discovery, especially for docs, GitHub repos, Stack Overflow, changelogs, companies, people, and similar-page retrieval.

Use Exa when: the agent needs to find relevant sources even when keywords are not obvious: libraries, docs, code examples, adjacent concepts, companies, people, technical research.

Pitfall: semantic indexes can underperform on exact freshness or exact SERP-style ranking. One practitioner writeup explicitly notes Exa’s strength in semantic similarity but flags freshness as a weakness in some tests.

Best cheap Google-style SERP JSON: Serper

Serper is the strongest “cheap Google results as JSON” option. It offers 2,500 free queries, no credit card, Google Search/Image/News/Maps/Places/Videos/Shopping/Scholar/Patents/Autocomplete endpoints, and paid packs from $1.00/1K down to $0.30/1K at high volume.

Use Serper when: you want Google-like search results cheaply and are willing to do your own extraction/reranking.

Pitfall: it is not agent-native context. You get SERP data; you still need a fetch/extract/context pipeline.

Best full-page extraction/crawling: Firecrawl

Firecrawl is more extraction/crawling infrastructure than pure search. It offers a free plan with 1,000 credits/month, then paid tiers such as Hobby and Standard, with concurrency increasing by tier.

In the AIMultiple benchmark, Firecrawl ranked second overall and did best when full-page context mattered; latency was slower than Brave/Tavily but still much faster than Parallel Pro.

Use Firecrawl when: the agent needs clean markdown/full-page content, not just snippets.

Pitfall: for discovery, pair it with Brave, Tavily, Exa, Serper, or your own query fan-out.

Best extraction utility: Jina Reader

Jina Reader is excellent as a modular URL-to-context layer. It converts URLs to LLM-friendly text, supports no-key/free/paid access tiers, and lists rate limits such as 20 RPM without API key, 500 RPM with free or paid API key, and 5,000 RPM premium for Reader API.

Use Jina when: you already have URLs and need cheap, simple markdown/text extraction.

Pitfall: Jina is not primarily a ranking/search decision engine; it is best as a fetch/extract component.

Best managed answer/search API: Perplexity

Perplexity now has several layers: Search API, Sonar API, Agent API, embeddings, URL fetch, and people/finance search tools. Its Search API is $5/1,000 requests with no token costs; Agent tool calls include web_search at $0.005/invocation and fetch_url at $0.0005/invocation; Sonar pricing combines token costs and request fees by search-context size.

Use Perplexity when: you want answer-style retrieval with citations, or a managed “research” experience.

Pitfall: if your agent needs raw chunks, custom ranking, or deterministic citation control, Perplexity can feel too opinionated.

Best business/premium-source angle: Linkup / Parallel / Valyu

Linkup gives 4,000 free queries, Fetch latency under 2s, Fetch pricing around $0.001–$0.005/request, and enterprise options for personalized indexes, dedicated refresh rates, private environment, ZDR, SOC 2 Type II, and SLA.

Parallel is notable for its CLI-first agent integration: its CLI covers search, extraction, enrichment, deep research, entity discovery, and monitoring, and its docs explicitly describe it as recommended for standalone agents.

Valyu is worth tracking for knowledge-work agents because it positions itself around web + proprietary/structured sources and deep research tasks, though it is less community-proven than Tavily/Exa/Brave/Serper.

Use these when: your use case is company research, sales intelligence, finance, professional research, academic/proprietary sources, or long-horizon deep research.

Pitfall: smaller community surface area, sometimes higher cost, and less battle-tested in general coding-agent workflows.

5. Important deprecations and traps

Bing Search API is effectively gone as a normal developer option. Microsoft says Bing Search APIs were retired on 11 August 2025, with existing instances decommissioned and no new customer signup; Microsoft points developers to Grounding with Bing Search in Azure AI Agents instead.

Google Custom Search JSON API is legacy-only. Google states existing customers have until 1 January 2027 to transition, the API is not available for new customers, and existing pricing remains 100 free queries/day plus $5/1,000 extra queries up to 10K/day.

Google SERP scraping got harder in 2025–2026. Proxyway reports Google’s SearchGuard and num=100 changes increased scraping costs/latency and disrupted scraper workflows, which matters if your agent stack depends on Google-SERP wrappers.

AI SEO / content spam is now a real retrieval-quality problem. One Reddit thread from r/RAG describes multiple APIs returning the same AI-SEO-optimized garbage for commercial/product queries, which matches a broader engineering reality: search API quality now depends heavily on source filtering, domain policies, and post-retrieval validation.

6. Practical decision guide
For a coding agent / Claude Code / Codex-like workflow

Use Brave or Exa for discovery, then Jina Reader or Firecrawl for extraction. If you want less plumbing, use Tavily first. Parallel is especially interesting if you like CLI-native tools because its CLI exposes search, extract, enrich, deep research, entity discovery, and monitoring.

For a cheap personal agent

Use Serper + Jina Reader or Brave + Jina Reader. Serper is cheaper if you want Google-like results; Brave is cleaner if you want an independent index and predictable raw web search.

For RAG over the live web

Use Tavily if you want a single API that returns agent-ready results. Use Firecrawl when the agent must ingest full pages cleanly. Use Exa when semantic discovery matters more than literal keyword matching.

For business intelligence, sales research, company enrichment

Test Linkup, Parallel, Perplexity Agent/Search, and Exa company/people search. Community reports specifically praise Linkup/Parallel for business-agent use cases, while Perplexity’s platform advertises Search, Agent, people search, finance search, and Sonar-style grounded answers.

For answer-first UX

Use Perplexity Sonar/Search, OpenAI Web Search, Anthropic Web Search, Gemini Grounding, or Brave Answers. These are good when you want a cited answer quickly, but weaker when you need full control over which sources were searched, how they were ranked, or which chunks went into context.

7. My recommended evaluation harness

For your own agents, do not choose based on generic benchmark scores. Run a small harness with 50–100 representative queries:

Discovery quality: Did the API find the right source, not just a plausible source?
Extraction quality: Could the agent get enough full text to answer?
Freshness: Did it find truly current pages for recent topics?
Noise resistance: Did it avoid SEO spam and vendor self-promotion?
Latency in loops: Multiply average latency by expected search calls per task.
Cost per solved task: Count searches, fetches, retries, summaries, and failed calls.
Agent ergonomics: Does the output require less prompt glue and post-processing?

The most robust architecture is usually multi-provider fallback: one cheap SERP provider, one agent-native semantic provider, and one extraction provider. A strong starting stack would be:

Tavily for fast agent-ready search, Brave or Serper as a cheaper/raw fallback, Exa for semantic/code-doc discovery, and Firecrawl/Jina for extraction.

That mix gives you coverage across the failure modes that matter most: stale results, keyword mismatch, SEO spam, extraction failure, and provider-specific outages.

Search snippets are not enough. Use:

search → fetch top 5–20 URLs → extract markdown → chunk → rerank → synthesize

Firecrawl, Jina Reader, and Crawl4AI are the strongest general-purpose tools for this stage.

7. Use browser automation sparingly

Browser automation should be triggered only when:

static fetch fails,
JS rendering matters,
login/session is required,
screenshots/visual state are needed,
the agent must operate a UI,
official API does not expose the data.

Otherwise, browser automation just adds latency, fragility, and safety risk.

10. Final recommendation for your agent tooling work

For your likely MCP/coding-agent use case, I would build around this:

1. Search:
   Brave + Exa + Tavily/Serper fallback

2. Extraction:
   Firecrawl + Jina Reader + Crawl4AI local fallback

3. Coding docs:
   Context7 or Ref.tools + GitHub + Stack Exchange

4. Community:
   Reddit + HN + GitHub issues/discussions + Telegram optional

5. Browser:
   Playwright MCP + Chrome DevTools MCP
   browser-use/Stagehand only when autonomous UI operation is needed

6. Validation:
   rerank, source-score, freshness-check, contradiction-check

The strongest quality improvement will not come from switching from Tavily to Exa or Brave to Serper. It will come from making the agent source-aware: official docs for APIs, GitHub/StackOverflow for code truth, Reddit/HN for user experience, Brave/Exa for discovery, Firecrawl/Jina/Crawl4AI for extraction, and browser automation only when the web cannot be read cleanly through APIs.
